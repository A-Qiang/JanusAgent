## Part 1 周总结

### 1. 智能销售多模态知识库感知底座立项与工程骨架落地

【本周进展】
为承接智能销售业务对文档、音视频等多模态知识入库的规模化需求，本周新建独立子包 `perception-agent`，作为多模态知识库的技术底座。核心动作包括：完成包配置与目录初始化（pyproject、README、gitignore），沉淀感知器核心抽象（`core/base.py`、`core/pipeline.py`）与文档导入器（`ingestors/document.py`）两条基础接口，配套输出 `scripts/download_model.py`（ModelScope 模型下载工具，支持版本选择、文件过滤、私有令牌）与 `test_mineru_model.py` 测试脚本，打通"模型下载 → 加载 → 推理验证"最小闭环。

【结果总结】
在智能销售侧从 0 到 1 建立了独立的**多模态感知工程基座**，让后续文档解析、模型微调、知识入库等工作有了明确的代码承载与扩展接口，避免了在业务系统中零散耦合。工程基座就绪意味着算法迭代与业务系统的发版节奏解耦，可以在离线环境高频、低成本验证多模态能力后再向生产同步。

### 2. 2025-2026 文档解析 / OCR 前沿论文系统性调研与技术路线选型

【本周进展】
为回答"智能销售场景下应该走大模型通用 VLM 还是专用轻量文档模型"这一关键选型问题，本周系统性调研了 2025-2026 年 11 篇文档解析 / 视觉 OCR 领域重要论文，产出三份沉淀：`VLM-wiki.md`（视觉语言模型基础）、`MinerU.md`（MinerU 深度剖析）、`论文调研.md`（11 篇论文分 **A 端到端 VLM / B 后处理 / C Agentic OCR / D 长文档架构**四类横向对比表，覆盖 MinerU2.5、dots.ocr、MonkeyOCR、PaddleOCR-VL、AgenticOCR、MACT、DeepSeek-OCR 2 等主流方案）。

【结果总结】
形成了智能销售多模态知识库的**技术路线判断依据**：明确选型 MinerU2.5-Pro-1.2B 作为文档解析基座（1-3B 轻量、二阶段解耦、数据引擎驱动），并识别出后处理层与 Agentic 化两个可组合的能力扩展方向。避免在方案选型阶段陷入"每篇论文都读一点、每个模型都试一试"的盲试消耗，把技术判断从"体感"升级为"横向可对标"。

## Part 2 价值焦点思考

本周围绕智能销售多模态知识库的三项工作，本质是在为"用大模型规模化承接业务侧非结构化知识"这条商业路径搭建**工程底座与判断力**，其价值可从以下三个维度审视：

**独立感知底座带来的商业敏捷性**：此前多模态相关能力若嵌在业务系统内，任何模型迭代都要绑定业务发版节奏、共享推理资源、承受环境约束。本周落地的 `perception-agent` 独立子包让感知能力从业务系统中解耦出来，形成可独立训练、独立部署、独立灰度的"感知服务层"。这为智能销售未来接入更多知识源（PDF 手册、音视频培训、图纸等）建立了可复用的收纳位，让每一次业务侧新知识引入都能以"加插件"而非"改主干"的方式完成，直接降低多模态知识入库的边际成本。

**轻量专用路线对应的直接商业价值**：11 篇论文横向对比后，明确选型 1-3B 参数的专用文档解析模型（MinerU2.5-Pro），而非 30B+ 的通用 VLM。这一判断的商业含义是——单卡即可承载生产推理、私有化部署门槛低、可按业务量弹性扩容，直接对应智能销售在客户侧的**推理成本可控性**与**私有化落地可行性**。数据驱动 + 二阶段解耦范式也意味着领域适配阶段能通过小数据量微调快速拿到收益，避免通用大模型的高微调成本陷阱。

**OKR 三层拆解沉淀出的团队协同资产**：把"文档解析模型微调"这一模糊技术命题拆解为「月度 OKR → 周级需求 → 每日任务卡 + 交付物 + 验收条件」的三层看板，本质是把不确定的算法探索封装成了确定性的工程节拍。这套模板同样可迁移到智能销售未来的语音理解、视频解析等模态方向，成为团队多模态攻关的**可复用项目管理方法论**，让新同事进项目就有清晰上下文，让跨周汇报有稳定的可对标基线。
