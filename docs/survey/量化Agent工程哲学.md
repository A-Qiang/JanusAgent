# 量化 Agent 工程哲学：Harness Engineering for Quants

> 本文档系统梳理「Harness Engineering（马具工程）」这一量化 Agent 设计哲学的核心概念，并结合 JanusAgent 的 `quant-agent` 包给出分阶段落地建议。
>
> 核心命题：**Agent = 模型 + Harness**。模型提供原始智能，可靠性则在 Harness（除模型外的一切）中被构建或摧毁。

---

## 1. 背景：每个量化团队都会撞上的那堵墙

2026 年，量化研究员在使用 AI Agent 时反复遭遇同一类问题：模型很聪明、提示词打磨得很精致、回测看起来也干净——然后 Agent 却改错了文件、写了一个窗口期错误的因子、幻觉出一个不存在的 API，或者悄悄产出了一个存在**前视偏差（look-ahead bias）**的策略。团队花三天调试一个十二步之前的工具调用，却没有任何痕迹可循。

**问题的根源几乎从来不是模型本身，而是模型周围的系统。**

Harness 的比喻来自给工作马配备的马具（缰绳、马鞍、嚼子）——用来引导一匹强大但不可预测的动物朝有用的方向前进：

- **AI 模型是马**：速度快，但没有方向
- **你是骑手**：负责设定目标与约束
- **Harness 是马具**：让这对搭档真正协同工作的一切

最有力的证据来自 LangChain：其编码 Agent 在 Terminal Bench 2.0 上从 Top 30 之外跃升至 Top 5（+13.7 个百分点，52.8% → 66.5%），**全程没有更换底层模型**（固定使用 GPT-5.2-Codex），只改变了三个变量——系统提示词、工具、中间件钩子。智能没有变，智能周围的环境全变了。

## 2. 概念演进时间线

| 时间 | 事件 |
|---|---|
| **2026-02-05** | HashiCorp 联合创始人 Mitchell Hashimoto 发博客给出定义："每当发现 Agent 犯了一个错误，就花时间工程化一个解决方案，确保这个错误永远不会再发生。" |
| **2026-02（几天后）** | OpenAI 发布《Harness Engineering: Leveraging Codex in an Agent-First World》，"Harness Engineering" 迅速成为 AI 工程领域核心词汇 |
| **持续** | LangChain 以「不换模型、只改 Harness」的方式在 Terminal Bench 2.0 提升 13.7 个百分点，成为标志性实证 |

### 与前序范式的边界

| 阶段 | 范式 | 关注点 | 适用场景 |
|---|---|---|---|
| 2023–2024 | **Prompt Engineering** | 优化单次交互的指令 | 一问一答 |
| 2025 中期 | **Context Engineering** | 系统级上下文设计（RAG / MCP / 记忆） | 上下文供给 |
| **2026** | **Harness Engineering** | 为 Agent 设计**完整运行环境** | 长链路、多 Agent、无人值守的生产工作流 |

量化交易台运行的不是单次交互，而是需要生成因子、编写回测代码、调用市场数据 API、跨市场环境评估性能、迭代模型架构、向共享研究仓库提交 PR 的长流程——可能持续数小时、跨多个 Agent、大部分步骤没有人类参与。Prompt / Context Engineering 都管不住这个，**只有 Harness Engineering 可以**。

## 3. 生产级 Harness 的 11 个原语

> 用户关注的十个核心概念（指令层、上下文管理、工具接口、执行环境、持久状态、编排、子 Agent、技能固化、验证门禁、Harness 演化）在下表中一一对应；加上「上下文交付」共 11 个原语。

### 原语 1：指令层（Instruction Layer）

告诉模型它是谁、做什么工作、必须遵守什么约束、绝不能做什么。

- OpenAI Codex 团队的教训：一个巨大的 `AGENTS.md` 会失败——上下文是稀缺资源，过大的指令文件会挤占任务和文档的空间，「太多指引 = 没有指引」。
- 修复方案：把 `AGENTS.md` 当成**目录**（约 100 行），指向结构化的 `docs/` 目录作为权威记录。
- 量化场景编码的是**结构性约束**（而非偏好）：使用特定供应商的复权收盘价、构建特征时不得使用目标变量衍生的特征、回测窗口必须包含至少一个熊市周期、所有因子假设必须在生成代码之前记录。

### 原语 2：上下文交付（Context Delivery）

Harness 给模型提供完成工作所需材料的机制。

- 类比：让研究员凭记忆写动量因子，vs. 把订单簿数据、收益率序列、既有因子库、基准规范都递到手上。

### 原语 3：上下文管理（Context Management）

问题不在于能否提供上下文，而在于**谁来决定此刻什么上下文重要**。

- 手段：从知识库检索相关因子研究的 RAG、优先排序最新市场环境数据的重排序、上下文窗口接近容量时的会话摘要、将长历史压缩为摘要而不丢失关键状态。

### 原语 4：工具接口（Tool Interface）

Agent 作用于世界的方式，MCP（Model Context Protocol）已成连接标准。

- 暴露对象：市场数据 API、回测引擎、因子数据库、风险模型、组合优化器、执行管理系统、版本控制。
- 核心设计原则：**幂等性 + 有界范围**。读取市场数据的工具是安全的；提交实盘订单的工具必须有**人类审批门禁**。

### 原语 5：执行环境（Execution Environment）

工具调用实际运行的地方——在什么约束、什么资源、什么信任级别下。

- 提示词里说「请不要碰生产密钥」依赖的是概率性合规；通过工程约束让 Agent **结构性地不可能**访问生产密钥，才是 Harness Engineering。
- 量化实践：每个研究会话都在干净环境中运行，固定数据、固定库版本，无法访问实盘持仓。

### 原语 6：持久状态（Persistent State）

Agent 做过什么、决定了什么、尝试过什么、学到了什么，必须存活于当前上下文窗口之外。

- 范例：微软 **RD-Agent(Q)**（NeurIPS 2025 接受）将量化研究分解为研究阶段与开发阶段，通过反馈阶段连接。Agent 记得自己试过什么、什么失败、为什么失败——**这才是迭代的引擎，而非模型的原始智能**。
- 实测数据：年化收益率约为经典因子库（Alpha158 / Alpha360）的 2 倍，使用因子数量减少 70% 以上，每次优化成本不到 10 美元。

### 原语 7：编排（Orchestration）

决定工作如何流动：生命周期钩子、心跳检测、重试、审批门禁、人类交接、工具排序、步骤序列化。

- 量化完整流程：生成假设 → 编写因子代码 → 运行单元测试 → 用清除交叉验证（purged CV）跑历史回测 → 与基准对比评估 → 记录结果到持久状态 → 若夏普比率超阈值则开 PR 供人审查，否则带失败原因返回假设生成。
- 关键：**每一步都有明确的成败标准，Agent 不能跳过步骤**。

### 原语 8：子 Agent（Sub-Agents）

复杂研究流程有分支：主 Agent 可能需同时探索动量因子、测试波动率调整变体、审阅现有文献。

- 实证：Anthropic 对比三 Agent 结构（Planner / Generator / Evaluator）与单 Agent——三 Agent 的 Harness 自动捕获了关键连接断裂的失败，单 Agent 则产出了「技术上可用但底层代码有问题」的输出。

### 原语 9：技能与程序（Skills & Procedures）— 技能固化

技能是 Harness 可调用的可复用、已命名的程序。

- 消除最危险的失败模式：Agent 从头重新发明程序，并在每次引入微妙错误。
- 回测技能每次都以相同方式运行清除交叉验证——相同封锁期、相同前向窗口、相同交易成本模型。**Agent 不允许在方法上即兴发挥。**

### 原语 10：验证与可观测性（Validation & Observability）— 验证门禁

Agent 说「我完成了」不是正确性的证据，**Harness 需要收据**。

- 量化验证：回测完成且无数据错误、因子在样本外与未来收益相关、测试集夏普比率高于底线、最大回撤在范围内、因子已通过前视偏差检测工具测试。
- 没有可观测性，调试错误结果就是「考古学」；有了它，就是「工程学」。

### 原语 11：Harness 演化（Harness Evolution）

正如 Hashimoto 的定义，这是一门迭代的纪律：**每一次 Agent 犯错，就在 Harness 中工程化一个永久性修复，让那个特定错误在结构上不可能重演。**

- OpenAI Codex 团队的教训：Codex 会复制仓库里已有的模式（哪怕是次优的）。他们最初每周五花 20% 时间清理「AI 烂代码」，后来把黄金原则直接编码进仓库，并建立后台清理流程（自动扫描偏差、更新质量等级、定向重构 PR）。
- 心法：**技术债像高息贷款——持续小额偿还。**

## 4. 量化金融中的六大失败模式与 Harness 修复

| 失败模式 | Harness 修复 |
|---|---|
| 因子构建中的前视偏差 | 执行环境强制数据 API 要求 `as_of_date` 参数 |
| 长研究会话中的上下文溢出 | 持久状态层在会话开始时检查原始假设 |
| 用微妙错误重新发明方法 | 将这些程序固化为已命名技能，Agent 不重新实现 |
| 静默工具失败 | 工具接口在返回 Agent 之前验证响应完整性 |
| 子 Agent 不一致 | 指令层与技能层统一编码数据标准 |
| 未经验证的完成 | 编排层设验证门禁：回测工具返回通过结果才能开 PR |

## 5. 结合 JanusAgent `quant-agent` 包的落地建议

### 5.1 当前现状

`packages/quant-agent`（理性之面）目前处于骨架阶段：

- `market.py`：`Bar` 数据类（symbol / timestamp / OHLCV）
- `strategies.py`：`Strategy` 数据类，`run()` 抛 `NotImplementedError`
- `__init__.py`：`hello()` 占位

技术栈：Python 3.12 + pydantic-ai + uv workspace，与 `companion-agent` 通过 Python import 而非 IPC 耦合。这意味着 Harness 的多数原语可直接以「pydantic-ai 工具/依赖注入 + 本地模块」形式落地。

### 5.2 11 原语 → quant-agent 落地映射

| 原语 | 在 quant-agent 中的落地形式 |
|---|---|
| 指令层 | 新建 `packages/quant-agent/QUANT_AGENTS.md`（约 100 行目录式），编码复权口径、禁用未来函数、回测窗口须含熊市等结构性约束 |
| 上下文交付 | 通过 pydantic-ai 的 `deps`（依赖注入）把 `Bar` 序列、因子规格、基准规范递给模型 |
| 上下文管理 | 因子研究知识库检索 + 会话摘要，避免长研究会话上下文溢出 |
| 工具接口 | 用 pydantic-ai `@agent.tool` 暴露：市场数据读取（幂等）、因子计算、回测执行、IC 计算；实盘下单类工具设人类审批门禁 |
| 执行环境 | 研究会话跑在固定数据版本、无实盘持仓访问权的沙箱；数据 API 强制 `as_of_date` 参数从源头杜绝前视偏差 |
| 持久状态 | 新增假设日志（hypothesis log）+ 回测结果日志，支撑跨会话迭代（参考 RD-Agent(Q)） |
| 编排 | 固化「假设→因子代码→单测→清除交叉验证回测→基准对比→记录→开 PR/回退」流水线，每步带成败标准 |
| 子 Agent | 拆分 Planner / Generator / Evaluator 三角色，借 agentpool 调度 |
| 技能固化 | 将清除交叉验证回测、IC 计算、前视偏差检测封装为命名技能，替换 `Strategy.run()` 的空实现，禁止即兴发挥 |
| 验证门禁 | 编排层加验证：回测无数据错误 + 样本外相关 + 夏普高于底线 + 回撤达标，方可开 PR |
| Harness 演化 | 每周 Harness 审查，把每次失败对应到某一层的永久修复 |

### 5.3 优先落地顺序（结合骨架现状）

1. **先立指令层与执行约束**：`QUANT_AGENTS.md` + 数据 API 的 `as_of_date` 强制约束——这是防前视偏差的第一道结构性防线，成本最低、收益最高。
2. **再补工具接口与技能固化**：把回测、IC 计算做成幂等工具/命名技能，让 `Strategy.run()` 有真正可复用的实现。
3. **最后接编排与验证门禁**：串起完整流水线，只有通过验证才能开 PR。

## 6. 8 周构建路线图

| 阶段 | 目标 |
|---|---|
| **第 1–2 周** | 基础：创建 `QUANT_AGENTS.md`，编码数据标准，搭建沙箱回测环境 |
| **第 3–4 周** | 工具接口：用 MCP 定义市场数据、因子计算、回测执行、IC 计算工具 |
| **第 5 周** | 持久状态与编排：构建假设日志与回测结果日志，连接完整编排流程 |
| **第 6 周** | 子 Agent 与技能：固化清除交叉验证回测、IC 计算、前视偏差检测等技能 |
| **第 7 周** | 验证与可观测性：加验证门禁与循环检测中间件，启用完整追踪日志 |
| **第 8 周及以后** | Harness 演化：每周审查「Agent 做错了什么？每个失败对应哪一层修复？」 |

## 7. 诊断清单：每个量化台都应该问的

当 AI Agent 产出错误结果时，本能反应是怪模型；**正确的本能是问：Harness 的哪一层走到了尽头？**

- 指令层缺少了数据标准规则？
- 上下文交付给了错误的因子规格？
- 上下文管理让原始策略约束被推出注意力窗口？
- 工具接口没有验证数据 API 响应？
- 执行环境让 Agent 访问了不该有的信息？
- 持久状态丢失了之前尝试过什么？
- 编排让 Agent 跳过了验证？
- 没有技能编码正确的回测方法？
- 验证层接受了自我报告的完成而没有真正运行回测？
- 没有追踪来识别十二个前置步骤中哪一步出了问题？

> **核心转变：从「模型犯了错误」到「模型周围的系统哪一层没能阻止这个错误」。**

对量化金融而言——一个错误因子在生产中消耗的是真金白银，研究管线中的系统性错误会在数百个策略中复利——这个区分不是哲学上的，**而是研究操作能否规模化与悄悄失败之间的区别。**
