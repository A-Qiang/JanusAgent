## 概述

LLM Wiki 是 Karpathy 提出的一种**利用 LLM 构建个人知识库的模式**。其核心思想是：让 LLM 不是在每次查询时从原始文档中重新检索知识（传统 RAG），而是**增量地构建和维护一个持久化的 wiki**——一组结构化、相互链接的 Markdown 文件，位于你和原始资料之间。

这份文档本身被设计为**直接复制粘贴给你的 LLM Agent**（如 OpenAI Codex、Claude Code、OpenCode 等），由 Agent 根据你的具体领域来落地实现。

---

## 核心思想：Wiki 是持久化、复利式的知识资产

### 传统 RAG 的问题

| 方面 | 传统 RAG | LLM Wiki |
|------|----------|----------|
| **知识获取** | 每次查询时从原始文档重新检索 | 一次性编译，持续维护 |
| **知识累积** | 无累积，每次从零开始 | 持续累积，每次添加源都让 wiki 更丰富 |
| **交叉引用** | 每次查询时临时拼接 | 交叉引用已预先建立 |
| **矛盾检测** | 无 | 矛盾已被标记 |
| **综合分析** | 每次重新综合 | 综合反映已读过的所有内容 |

### 关键区别

> **Wiki 是一个持久化、复利式的资产。** 交叉引用已经存在，矛盾已被标记，综合分析已反映了你读过的一切。每添加一个源、每问一个问题，wiki 都变得更丰富。

### 人机分工

| 角色 | 职责 |
|------|------|
| **人类** | 策划信息源、探索方向、提出好问题、思考意义 |
| **LLM** | 摘要、交叉引用、归档、记账——所有让知识库随时间变得有用的繁杂工作 |

> 你几乎不（或很少）自己写 wiki——LLM 编写并维护全部内容。实践中，一侧开着 LLM Agent，另一侧开着 Obsidian，LLM 根据对话编辑 wiki，你实时浏览结果——跟随链接、查看图谱视图、阅读更新后的页面。**Obsidian 是 IDE，LLM 是程序员，wiki 是代码库。**

## 适用场景

| 场景 | 说明 |
|------|------|
| **个人成长** | 追踪目标、健康、心理、自我提升——归档日记、文章、播客笔记，构建关于自己的结构化画像 |
| **研究** | 数周/数月深入某个主题——阅读论文、文章、报告，增量构建带有演化论点的综合 wiki |
| **读书** | 逐章归档，构建角色、主题、情节线索及其关联的页面。类似 Tolkien Gateway 等粉丝 wiki，但由 LLM 完成交叉引用和维护 |
| **商业/团队** | LLM 维护的内部 wiki，输入 Slack 线程、会议记录、项目文档、客户电话。人类在环审查更新 |
| **其他** | 竞品分析、尽职调查、旅行规划、课程笔记、爱好深挖——任何需要随时间累积知识并组织化的场景 |

## 架构：三层结构

```
┌─────────────────────────────────────────────────┐
│              Raw Sources（原始源）                │
│  文章、论文、图片、数据文件                         │
│  → 不可变，LLM 只读不写                            │
│  → 这是你的真相来源（source of truth）              │
├─────────────────────────────────────────────────┤
│              The Wiki（知识 wiki）                │
│  LLM 生成的 Markdown 文件目录                      │
│  → 摘要、实体页、概念页、对比、概述、综合            │
│  → LLM 完全拥有此层：创建、更新、交叉引用、保持一致   │
│  → 你读它；LLM 写它                                │
├─────────────────────────────────────────────────┤
│              The Schema（模式配置）                │
│  如 CLAUDE.md / AGENTS.md                        │
│  → 告诉 LLM wiki 的结构、约定和工作流程             │
│  → 使 LLM 成为纪律严明的 wiki 维护者，而非通用聊天机器人 │
│  → 你和 LLM 共同演化此文件                         │
└─────────────────────────────────────────────────┘
```

### 三层详解

**1. Raw Sources（原始源）**
- 你策划的源文档集合：文章、论文、图片、数据文件
- **不可变**——LLM 读取但不修改
- 这是真相来源

**2. The Wiki（知识 wiki）**
- LLM 生成的 Markdown 文件目录
- 包含：摘要、实体页、概念页、对比、概述、综合分析
- LLM **完全拥有**此层：创建页面、新源到来时更新、维护交叉引用、保持一致性
- 你读它；LLM 写它

**3. The Schema（模式配置）**
- 一个文档（如 Claude Code 的 `CLAUDE.md`、Codex 的 `AGENTS.md`）
- 告诉 LLM：wiki 如何组织、约定是什么、摄入源/回答问题/维护 wiki 时遵循什么工作流
- **关键配置文件**——使 LLM 成为纪律严明的 wiki 维护者
- 你和 LLM 随时间共同演化此文件

## 三大操作

### 1. Ingest（摄入）

```
用户投放新源 → LLM 读取源 → 与用户讨论关键要点
→ LLM 编写摘要页 → 更新索引
→ 更新相关实体页和概念页（可能触及 10-15 个页面）
→ 追加日志条目
```

- **推荐方式**：逐个摄入，保持参与——阅读摘要、检查更新、引导 LLM 关注重点
- **也可批量摄入**：一次处理多个源，减少监督
- 将适合你的工作流记录在 schema 中

### 2. Query（查询）

```
用户提问 → LLM 搜索相关页面 → 读取并综合答案（带引用）
→ 答案可以是：Markdown 页面 / 对比表 / 幻灯片(Marp) / 图表(matplotlib) / 画布
```

**关键洞察**：好的答案应该**回填为 wiki 新页面**。你请求的对比、发现的分析和联系——这些是有价值的，不应消失在聊天历史中。这样你的探索就像摄入的源一样在知识库中复利增长。

### 3. Lint（健康检查）

定期让 LLM 检查 wiki 健康：

- [ ] 页面之间的矛盾
- [ ] 被更新源取代的过时声明
- [ ] 没有入站链接的孤儿页面
- [ ] 被提及但缺少独立页面的重要概念
- [ ] 缺失的交叉引用
- [ ] 可通过 web 搜索填补的数据空白

LLM 擅长建议新的调查问题和新的源——这保持 wiki 随其增长而健康。

## 索引与日志

两个特殊文件帮助导航不断增长的 wiki：

### index.md — 内容导向

| 属性 | 说明 |
|------|------|
| **性质** | 内容目录，wiki 中所有内容的目录 |
| **内容** | 每个页面：链接 + 一行摘要 + 可选元数据（日期、源数量） |
| **组织** | 按类别（实体、概念、源等） |
| **更新** | 每次摄入时更新 |
| **查询用途** | LLM 回答查询时先读索引找到相关页面，再深入阅读 |
| **适用规模** | ~100 个源、~数百个页面时表现良好，无需嵌入式 RAG 基础设施 |

### log.md — 时间导向

| 属性 | 说明 |
|------|------|
| **性质** | 追加式时间记录 |
| **内容** | 摄入、查询、lint 记录 |
| **格式建议** | 统一前缀，如 `## [2026-04-02] ingest \| Article Title` |
| **可解析性** | 可用 unix 工具解析，如 `grep "^## \[" log.md \| tail -5` 获取最近 5 条 |
| **用途** | wiki 演化的时间线；帮助 LLM 理解最近做了什么 |

## 可选 CLI 工具

当 wiki 增长到一定规模，可以构建小工具帮助 LLM 更高效地操作 wiki：

### 搜索引擎：qmd

- **项目**: [qmd](https://github.com/tobi/qmd)
- **功能**: Markdown 文件的本地搜索引擎
- **特性**: 混合 BM25/向量搜索 + LLM 重排序，全部在设备端
- **接口**: 同时提供 CLI（LLM 可 shell 调用）和 MCP Server（LLM 可作为原生工具使用）
- **适用场景**: wiki 规模增长后，index 文件不再够用时使用

> 也可以让 LLM 帮你 vibe-code 一个简单的搜索脚本——按需构建即可。

## 技巧与窍门

### Obsidian 相关

| 技巧 | 说明 |
|------|------|
| **Web Clipper** | Obsidian 浏览器扩展，将网页文章转为 Markdown，快速获取源材料 |
| **图片本地化** | Settings → Files and links → 设置 "Attachment folder path" 为 `raw/assets/`；Hotkeys 绑定 "Download attachments for current file"（如 Ctrl+Shift+D），剪藏文章后一键下载所有图片到本地 |
| **图片读取注意** | LLM 无法一次性读取带内联图片的 Markdown——变通方法是先读文本，再单独查看引用的图片 |
| **Graph View** | 观察 wiki 形态的最佳方式——什么连着什么、哪些页面是枢纽、哪些是孤儿 |
| **Marp** | 基于 Markdown 的幻灯片格式，Obsidian 有插件，可直接从 wiki 内容生成演示文稿 |
| **Dataview** | Obsidian 插件，对页面 frontmatter 运行查询。如果 LLM 在 wiki 页面添加 YAML frontmatter（标签、日期、源数量），Dataview 可生成动态表格和列表 |

### 版本控制

> Wiki 就是一个 Markdown 文件的 git 仓库。你免费获得版本历史、分支和协作能力。

## 为什么有效

### 核心论点

> 维护知识库的乏味部分不是阅读或思考——而是记账。更新交叉引用、保持摘要最新、注意新数据何时与旧声明矛盾、维护数十个页面的一致性。**人类放弃 wiki 是因为维护负担增长快于价值。**

### LLM 的优势

- LLM 不会无聊
- LLM 不会忘记更新交叉引用
- LLM 可以一次性触及 15 个文件
- **Wiki 保持维护状态，因为维护成本接近于零**

### 与 Memex 的关联

这个想法在精神上与 Vannevar Bush 1945 年提出的 **Memex** 相关——一个个人化的、策划式知识存储，文档之间有关联路径。Bush 的愿景比互联网实际成为的样子更接近这个想法：私有的、主动策划的，文档之间的联系与文档本身一样有价值。

> Bush 无法解决的部分是谁来做维护。LLM 解决了这个问题。

## 设计哲学：有意抽象

文档末尾强调：

> 本文档有意保持抽象。它描述的是想法，不是具体实现。确切的目录结构、schema 约定、页面格式、工具——所有这些都取决于你的领域、偏好和 LLM 选择。上面提到的一切都是可选和模块化的——选择有用的，忽略没用的。

**使用方式**：将此文档分享给你的 LLM Agent，共同实例化一个适合你需求的版本。文档的唯一职责是传达模式，其余的由你的 LLM 来解决。

## 与 JanusAgent 项目的关联

LLM Wiki 模式与 JanusAgent 中的以下方面有直接关联：

1. **perception-agent**: VLM-wiki 项目已基于此思想扩展为全模态版本，支持图片/视频/音频输入
2. **知识管理**: `.agent/memory` 和 `.agent/context` 目录可借鉴此模式进行结构化知识管理
3. **Agent 技能沉淀**: survey 目录本身就是一个轻量版 wiki，记录团队的技术调研成果
4. **Schema 驱动**: 类似 `AGENTS.md` / `CLAUDE.md` 的模式配置理念已在项目 `.agent` 结构中体现

## 总结一句话

> **让 LLM 做知识库的记账工作（摘要、交叉引用、归档、一致性维护），人类专注策划信息源和提出好问题——wiki 因维护成本趋近于零而持续复利增长。**
