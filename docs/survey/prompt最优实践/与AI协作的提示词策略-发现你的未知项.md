# 与 AI 协作的提示词策略：发现你的「未知项」

> 本文总结自 Claude Code 工程师 Thariq Shihipar 的博客《A Field Guide to Fable: Finding Your Unknowns》。
> 核心洞见：**人和模型之间存在信息差**——用户提供的提示词、Skill、上下文，与任务实际执行之间的差异。高质量协作的关键，在于主动发现并澄清这些差异。

- **原文（英文博客）**：https://x.com/trq212/status/2073100352921215386
- **示例合集**：https://thariqs.github.io/html-effectiveness/unknowns/

## 一、核心隐喻：地图 ≠ 疆域

- **地图**：对待完成工作的表征，即你交给模型的提示词、技能、上下文。
- **疆域**：真正需要发生工作的地方——代码库、现实世界及其中真实存在的约束。
- **未知项（Unknowns）**：地图与疆域之间的差距。当模型遇到未知项时，它只能根据对你意图的「最佳猜测」来做决定。工作越复杂，未知项越多。

> 工作质量的瓶颈，取决于你澄清未知项的能力。仅仅提前规划并不总是足够，未知项可能在实现深处才浮现，甚至指向「应该换一种完全不同的解法」。与模型协作本质上是**在实现前 / 实现中 / 实现后不断发现未知项的迭代过程**。

---

## 二、未知项的四象限

带着问题找模型时，可从四个角度拆解：

| 象限 | 含义 |
| --- | --- |
| **已知的已知** | 我明确写在提示词里的内容——我告诉智能体我想要什么。 |
| **已知的未知** | 我知道自己还没弄清楚的事。 |
| **未知的已知** | 对我显而易见、以至于懒得写下来，但一看到就能认出来的东西。 |
| **未知的未知** | 完全没考虑过、甚至不知道自己不知道的事。 |

> 优秀的智能体式程序员往往拥有**更少的未知项**：清楚自己想要什么、细节充分、与代码库和模型行为高度同步。但他们也**预设未知项的存在**。减少并提前规划未知项，就是智能体式编程的核心能力。

## 三、协作原则：帮助模型以帮助自己

给模型下指令是一种微妙的平衡：

- **过于具体** → 即使有更好的方案，模型也会死板照做。
- **过于模糊** → 模型会基于「行业最佳实践」自行假设，而这些未必适合你的任务。

**关键做法**：给模型足够的起点上下文——
- 告诉它你目前思考到哪一步；
- 说明你对问题和代码库的熟悉程度；
- 让它像一个「思考伙伴」一样与你协作。

模型能极快搜索代码库和互联网、在多数主题上掌握比你更多的通用知识、也能更快从失败中迭代——用它来加速发现未知项。

## 四、实操模式（按阶段）

### 实现前（Before）

**1. 盲点扫描（Blindspot Pass）**
适用于进入不熟悉的新模块 / 新领域，「未知的未知」很多时。直接使用「blindspot pass」「unknown unknowns」这些字眼，并告诉模型你是谁、你已知什么。

> 提示词示例：
> - 「我正在添加一个新的身份认证提供方，但我对这个代码库的认证模块一无所知。你能不能做一次 blindspot pass，帮我找出相关的 unknown unknowns，并帮助我更好地给你写提示词？」
> - 「我不了解调色，但需要给这个视频做调色。你能不能教我理解自己在调色方面的 unknown unknowns，这样我就能写出更好的提示词？」

**2. 头脑风暴与原型（Brainstorm & Prototype）**
适用于「未知的已知」多的领域（看到才知道标准的事）。尽早在原型阶段暴露它们——越晚发现，回退代价越大。原型可用假数据、不接后端。

> 提示词示例：
> - 「我想为这组数据做一个仪表盘，但我没什么视觉品味。帮我做一个 HTML 页面，给出 4 种风格差异很大的设计方向，好让我根据结果做反馈。」
> - 「在真正接线之前，先用假数据做一个单独的 HTML 文件，模拟新的编辑器工具栏。我想先对布局做反馈，你再去动真实应用。」
> - 「这是我的粗略问题：用户完成 onboarding 后流失。搜索代码库，头脑风暴 10 个可介入的点，从最低成本到最有野心都列出来。」

**3. 反问 / 采访（Interview）**
头脑风暴后仍有未知项时，让模型围绕歧义点采访你。

> 提示词示例：
> - 「请一次只问我一个问题，围绕任何存在歧义的地方采访我。优先提出那些我的回答会改变架构设计的问题。」

**4. 参考资料（References）**
当你无法用语言精确描述想要的东西时，用参考物代替。**最好的参考资料是源代码**——直接把模型指向对应文件夹并说明该看什么，即使参考代码是另一种语言也没关系。

> 提示词示例：
> - 「vendor/rate-limiter 里的这个 Rust crate 实现的正是我想要的回退重试行为。请阅读它，并在我们的 TypeScript API 客户端中重新实现相同的语义。」

**5. 实现计划（Implementation Plan）**
准备开始实现时，先让模型整理一份计划供审阅，**把最可能变动的决策点（数据模型、类型接口、UX 流程）放在开头**。

> 提示词示例：
> - 「用 HTML 写一份实现计划，但开头先呈现我最可能修改的决策点：数据模型变更、新的类型接口，以及任何面向用户的内容。机械性的重构放在最下面。」

### 实现中（During）

**6. 实现笔记（Implementation Notes）**
无论规划多充分，总有未知的未知潜伏。让模型维护一个临时的 `implementation-notes.md`（或 `.html`），记录偏离决策，供下一次迭代学习。

> 提示词示例：
> - 「请维护一个 `implementation-notes.md`。如果遇到某个边界情况导致你必须偏离原计划，请选择保守方案，在 'Deviations' 下记录原因，然后继续推进。」

### 实现后（After）

**7. 推介与解释文档（Pitch & Explainer）**
发布时，最重要的是获得他人理解、支持与批准。生成推介 / 解释类工件，可加速评审者理解、加速专家审批。

> 提示词示例：
> - 「把原型、规格说明和实现笔记打包成一个单独文档，我可以直接发到 Slack 争取支持。开头先放演示 GIF。」

**8. 测验（Quiz）**
只看 diff 往往只能浅层理解。让模型给你足够上下文后再围绕本次变更**出题考你**，通过测验后再合并。

> 提示词示例：
> - 「我想确保自己理解了这次变更里发生的一切。请给我一份 HTML 报告，帮助我阅读理解变更（上下文、直觉解释、具体做了什么），并在底部附一份我必须通过的测验。」

## 五、串联案例：Fable 发布视频（由 Claude Code 剪辑）

作者在一个全新领域（视频剪辑）中串联使用上述方法：

1. **从已知出发**：知道模型能用代码剪辑 / 转录视频，但不确定准确度 → 让模型解释 Whisper 转录原理，确认能否用 ffmpeg 剪掉语气词和停顿。
2. **原型验证**：想要 UI 与语音时间同步但不确定可行 → 让模型用 Remotion + 转录文本做视频原型验证想法。
3. **发现未知的未知**：视频灰暗是调色问题，但自己不懂「什么算好」→ 放弃盲目生成多版本，改为**让模型教自己调色**，从而发现未知项。

## 六、结论：让地图与疆域匹配

- 模型越强，用对方法就能完成越多事情。
- 当一个长周期任务返回错误结果时，通常说明你需要花更多时间**定义未知项**，或创建一份让模型能灵活应对的实现计划。
- 每一份解释文档、头脑风暴、访谈、原型、参考资料，都是**在修复代价变高之前，低成本地发现你原本不知道的事情**的手段。

> 开始下一个项目时，先让模型帮你找到自己的未知项。

## 附：可直接套用的提示词速查

| 阶段 | 目的 | 关键触发语 |
| --- | --- | --- |
| 实现前 | 找盲点 | 「做一次 blindspot pass，找出 unknown unknowns」 |
| 实现前 | 探方向 | 「给出 4 种风格差异很大的设计方向」/「头脑风暴 10 个方案」 |
| 实现前 | 消歧义 | 「一次只问一个问题，优先问会改变架构的问题」 |
| 实现前 | 对齐意图 | 「阅读这个文件夹 / crate，重新实现相同语义」 |
| 实现前 | 定计划 | 「实现计划开头先列最可能修改的决策点」 |
| 实现中 | 留痕迹 | 「维护 implementation-notes.md，在 Deviations 下记录偏离」 |
| 实现后 | 争支持 | 「打包成一个文档，开头放演示 GIF」 |
| 实现后 | 验理解 | 「给我 HTML 报告 + 底部测验，通过后我才合并」 |
