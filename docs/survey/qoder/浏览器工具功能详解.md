# Qoder 浏览器工具（browser-use）功能详解

> 本文系统梳理我在上一轮任务中调用的「浏览器工具」的全部能力。该工具以 MCP Server（`browser-use`）形式提供，底层驱动一个真实的 Chromium 内核浏览器（基于 Chrome DevTools Protocol），因此能像真人一样打开网页、点击、输入、截图，并读取网络与控制台信息。
>
> 它与普通「网页抓取（WebFetch）」的本质区别在于：**会真正执行 JavaScript 并渲染页面**，所以能拿到动态加载、需要登录或有反爬验证的内容——这也是上次 X 和微信文章能被读取的原因。

---

## 一、整体定位

- **调用形式**：既可由我直接调用底层 MCP 工具，也可以派发一个 **Browser 子代理（Browser Agent）** 来自主完成「导航 → 等待 → 操作 → 抓取」的完整流程。
- **能力本质**：一个「被 AI 操控的真实浏览器」，共 **16 个原子工具**，可分为五大类。
- **核心工作范式**：多数操作依赖 `take_snapshot` 生成的**元素 uid**——先「快照」拿到页面结构与元素标识，再用 uid 精确定位点击/输入目标。这比截图识别更稳定、更省资源。

---

## 二、能力全景（16 个工具分五类）

### 类别 1：页面与标签页管理（导航控制）

| 工具 | 功能 | 关键参数 |
| --- | --- | --- |
| `list_pages` | 列出浏览器中当前打开的所有页面（标签页） | 无 |
| `select_page` | 选定某个页面作为后续操作的上下文 | `pageIdx`（页面索引） |
| `navigate_page` | 在当前页面执行导航 | `type`：`url`/`back`/`forward`/`reload`；`url`；`ignoreCache`；`timeout` |

> 说明：`navigate_page` 一个工具即涵盖「打开新网址、后退、前进、刷新」四种动作；`ignoreCache` 可强制忽略缓存重新加载。

### 类别 2：页面内容感知（读取页面）

| 工具 | 功能 | 关键参数 |
| --- | --- | --- |
| `take_snapshot` | 基于无障碍树（a11y tree）生成页面的**文本快照**，列出所有元素及其唯一 uid | `filePath`（可存文件）；`verbose`（详细模式） |
| `take_screenshot` | 对整页或某个元素截图 | `format`（png/jpeg）；`fullPage`（整页）；`quality`；`uid`（指定元素）；`filePath` |

> **官方建议优先用快照而非截图**：快照是结构化文本，AI 更易解析且能拿到可点击的 uid；截图仅用于需要「视觉确认」的场景（如布局、样式、验证码）。

### 类别 3：页面交互（模拟人操作）

| 工具 | 功能 | 关键参数 |
| --- | --- | --- |
| `click` | 点击元素（支持双击） | `uid`；`dblClick` |
| `hover` | 悬停到元素上（触发悬浮菜单等） | `uid` |
| `fill` | 向输入框/文本域填值，或选择 `<select>` 下拉项 | `uid`；`value` |
| `drag` | 把一个元素拖拽到另一个元素上 | `from_uid`；`to_uid` |
| `press_key` | 按下单个键或组合键（如 `Enter`、`Control+A`、`Control+Shift+R`） | `key` |
| `upload_file` | 通过文件输入元素上传本地文件 | `uid`；`filePath` |
| `handle_dialog` | 处理浏览器弹窗（alert/confirm/prompt） | `action`（accept/dismiss）；`promptText` |

> 交互要点：`fill` 适合常规输入；键盘快捷键、导航键、特殊组合键必须用 `press_key`（多次按键要多次调用）；原生弹窗会阻塞页面，需 `handle_dialog` 接受或取消。

### 类别 4：等待与同步（应对动态加载）

| 工具 | 功能 | 关键参数 |
| --- | --- | --- |
| `wait_for` | 等待指定文本出现在页面上 | `text`；`timeout` |

> 用于应对异步渲染：先 `navigate_page` 打开页面，再 `wait_for` 等关键文字出现，确保内容加载完成后再抓取，避免拿到半成品页面。

### 类别 5：底层调试与脚本执行（高级能力）

| 工具 | 功能 | 关键参数 |
| --- | --- | --- |
| `evaluate_script` | 在页面内执行一段 **JavaScript 函数**，返回 JSON 可序列化结果，可传入元素 uid 作参数 | `function`（JS 函数声明）；`args`（元素 uid 数组） |
| `list_network_requests` | 列出上次导航以来的所有网络请求，可按资源类型过滤 | `resourceTypes`（document/xhr/fetch/script/image…）；`pageIdx`；`pageSize`；`includePreservedRequests` |
| `list_console_messages` | 列出页面控制台消息，可按级别过滤 | `types`（log/info/warn/error/…）；`pageIdx`；`pageSize`；`includePreservedMessages` |

> 这三者是最强的「工程级」能力：
> - `evaluate_script` 可直接抽取 DOM 文本、操作页面状态、发起 fetch，是抓取复杂内容的终极手段；
> - `list_network_requests` 能观察接口调用（常用于逆向 API、排查加载失败）；
> - `list_console_messages` 能捕获前端报错，非常适合调试 Web 应用。

## 三、典型工作流

### 场景 A：抓取需 JS 渲染 / 有反爬的文章（即上次任务）
```
navigate_page(url)          # 打开目标文章
  → wait_for(关键标题文本)   # 等页面渲染完成
  → take_snapshot()          # 拿到结构化正文
  →（若有登录/验证弹窗）handle_dialog 或 click 关闭
  → 滚动 / evaluate_script    # 补全长文全文
```

### 场景 B：自动化表单操作
```
take_snapshot() → fill(用户名 uid) → fill(密码 uid) → click(登录按钮 uid) → wait_for(登录成功标志)
```

### 场景 C：调试一个本地 Web 应用
```
navigate_page(localhost)
  → list_console_messages(types=[error])  # 看前端报错
  → list_network_requests(resourceTypes=[xhr,fetch])  # 看接口调用
  → take_screenshot()  # 视觉确认渲染结果
```

## 四、能力边界与注意事项

1. **uid 会随快照更新**：每次页面变化后应重新 `take_snapshot`，始终使用最新的 uid，否则定位失效。
2. **优先快照、慎用截图**：截图开销大且不含可操作 uid，仅在需要视觉判断时使用。
3. **动态内容需显式等待**：不要在 `navigate_page` 后立即抓取，配合 `wait_for` 更稳。
4. **可突破 JS/登录/验证限制**：真实浏览器渲染是它相对纯 HTTP 抓取的最大优势，但涉及登录/验证码时可能仍需人工介入。
5. **`evaluate_script` 返回值必须 JSON 可序列化**：无法直接返回 DOM 节点，需在函数内转成文本/对象。

## 五、一句话总结

Qoder 的浏览器工具 = **一个可被 AI 精确操控的真实 Chromium 浏览器**：
- **看得见**（快照 + 截图）
- **动得了**（点击/输入/拖拽/上传/按键/弹窗）
- **等得住**（wait_for 同步动态内容）
- **钻得深**（执行 JS、抓网络请求、读控制台）

因此它既能当「智能网页阅读器」（抓取动态/反爬内容），也能当「Web 自动化机器人」与「前端调试助手」。
