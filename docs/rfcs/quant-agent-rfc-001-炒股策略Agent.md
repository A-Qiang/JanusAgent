## 1. 摘要

本 RFC 提出一个基于用户既有炒股逻辑的量化交易 Agent 设计方案。核心是把用户的**「涨卖跌买 + 适当追涨」双策略**与选股范围形式化为 Agent 可执行的参数化规则，并套用 Harness Engineering 的 11 个原语，用**结构性约束**保证交易纪律与可靠性——而非依赖模型「自觉遵守」。

核心命题延续项目哲学：**Agent = 模型 + Harness**。策略正确性由 Harness（规则引擎、风控门禁、验证层）保证，模型只负责选股研判与信号生成等「需要智能」的环节。

---

## 2. 背景与动机

`packages/quant-agent` 目前仅有骨架（`Bar` 数据类、`Strategy.run()` 空实现）。用户已明确一套经过实践的炒股逻辑，需要将其沉淀为可自动化执行、可回测、可风控的 Agent 系统。

关键诉求：
- 把「凭经验判断」的买卖点，转化为**可复现的参数化规则**；
- 用 Harness 把交易纪律（如「单批不超过 10w」「卖出即清仓」）变成**结构上不可违反的约束**；
- 保留人类对高风险操作（如追涨 all in、实盘下单）的**审批门禁**。

---

## 3. 目标与非目标

### 目标（In Scope）
- 形式化用户的双策略买卖规则与仓位管理逻辑；
- 定义选股范围（行业 / 板块白名单）与候选池生成机制；
- 设计基于 Harness 11 原语的系统架构；
- 提供可回测的信号 → 决策 → 风控 → 执行流水线。

### 非目标（Out of Scope，本期不做）
- 实盘自动下单（本期仅到「生成交易建议 + 人类审批」为止）；
- 高频 / 日内交易（本策略为波段 / 中线）；
- 复杂衍生品、杠杆、融券。

---

## 4. 炒股逻辑形式化（核心）

> 将用户口述规则转为精确、参数化、机器可执行的定义。所有阈值均设为**可配置区间**，具体取值由参数表（第 8 节）统一管理。

### 4.1 策略 A：涨卖跌买（均值回归 / 波段）

**买入（分批加仓）**

- 参考价：`P_high_2m` = 标的近两个月（约 60 个交易日）滚动最高价
- 买入触发：`current_price ≤ P_high_2m × buy_ratio`，其中 `buy_ratio ∈ [0.60, 0.70]`
- 单批金额：`single_buy ∈ [1w, 2w]`（典型），硬上限 `single_buy ≤ 10w`
- 加仓方式：低于预期时分批买入（非一次性）

**卖出（清仓）**

- 收益率：`roi = (current_price − avg_cost) / avg_cost`
- 卖出触发：`roi ∈ [0.30, 0.50]`
- 卖出方式：**直接清仓**（一次性卖出全部持仓）

### 4.2 策略 B：适当追涨（动量）

**买入（all in）**

- 触发条件：标的「预期较高」（信号定义见 §4.4，**待明确**）
- 建仓方式：**all in**（受组合级仓位上限约束，见 §7 风控）

**卖出**

- 卖出触发：`roi ∈ [0.50, 1.00]`
- 卖出方式：直接清仓

### 4.3 仓位管理规则

| 场景 | 建仓方式 | 约束 |
|---|---|---|
| 涨卖跌买 - 买入 | 分批，1-2w/批 | 单批 ≤ 10w（硬约束） |
| 适当追涨 - 买入 | all in | 受组合单标的上限约束 |
| 任意卖出 | 一次性清仓 | 无剩余持仓 |

### 4.4 待明确点（Open Questions）

- **「高预期」信号如何量化？** 追涨的入场信号目前是主观判断，需定义可计算指标（如放量突破、板块联动、动量因子排名等）。
- **区间内具体触发点如何选？** `buy_ratio` 落在 0.60–0.70、卖出 `roi` 落在 0.30–0.50，具体取哪个值——是固定值、按波动率自适应，还是模型给建议？
- **两个月窗口**按自然月还是交易日计？默认按交易日（~60 日），可配置。

## 5. 选股范围

**行业白名单**（朝阳产业 / 高新技术）：
- AI、具身智能

**重点板块（sector 白名单）**：
- 光通信
- 算力
- 存储
- 机器人

**候选池生成**：
- 由「行业 + 板块白名单」过滤全市场，形成候选池 `universe`；
- 只有 `universe` 内标的才进入策略信号计算（**指令层结构性约束**：Agent 不得交易白名单外标的）。

## 6. 系统设计：映射 Harness 11 原语

| 原语 | 在本 Agent 中的落地 |
|---|---|
| **指令层** | `QUANT_AGENTS.md` 编码：只交易白名单板块、单批 ≤10w、卖出必清仓、禁止杠杆等结构性铁律 |
| **上下文交付** | 通过 pydantic-ai `deps` 注入行情（`Bar` 序列）、持仓、成本、`P_high_2m` 等 |
| **上下文管理** | 候选池筛选后只保留相关标的上下文，避免全市场噪声 |
| **工具接口** | `@agent.tool` 暴露：行情读取（幂等）、最高价计算、ROI 计算、信号生成、下单建议（需审批） |
| **执行环境** | 沙箱回测环境，固定数据版本；实盘密钥结构性隔离，Agent 无法直接下单 |
| **持久状态** | 持仓账本、成本记录、每笔交易决策日志（买卖点、触发规则、当时价格） |
| **编排** | 选股 → 信号 → 决策 → 风控门禁 → 交易建议 的固定流水线（§7） |
| **子 Agent** | 选股 Agent / 策略信号 Agent / 风控 Agent 三角色，借 agentpool 调度 |
| **技能固化** | 「近两月最高价」「ROI 计算」「仓位校验」封装为命名技能，禁止即兴实现 |
| **验证门禁** | 下单建议须通过：白名单校验 + 单批金额校验 + 组合仓位校验，方可输出 |
| **Harness 演化** | 每次策略失误对应到某一层的永久修复（如误买白名单外标的 → 加强指令层+工具层校验） |

---

## 7. 编排流程

```
1. 选股（Universe 构建）
   └─ 行业+板块白名单过滤 → 候选池
2. 信号计算（对候选池每只标的）
   ├─ 策略A：current_price 与 P_high_2m×buy_ratio 比较 → 买入信号
   ├─ 策略A：持仓 roi ∈ [0.30,0.50] → 清仓信号
   └─ 策略B：高预期信号命中 → all in 信号；roi ∈ [0.50,1.00] → 清仓信号
3. 决策合成
   └─ 汇总各标的信号，生成候选交易列表
4. 风控门禁（验证层，硬约束）
   ├─ 白名单校验：标的必须在 universe 内
   ├─ 单批校验：涨卖跌买单批 ≤ 10w
   ├─ 组合校验：单标的仓位/总仓位不超限
   └─ 追涨 all in：需人类审批门禁
5. 输出交易建议 + 记录持久状态
   └─ 达标 → 生成交易建议供人审批；不达标 → 记录拒绝原因
```

**每一步都有明确成败标准，Agent 不能跳过风控门禁。**

## 8. 参数配置表

> 遵循项目「按包放置 .env + 环境变量注入」的轻量配置约定，策略阈值集中管理，便于回测调参。

| 参数 | 含义 | 默认值 | 可调范围 |
|---|---|---|---|
| `HIGH_PRICE_WINDOW` | 最高价回看窗口 | 60（交易日） | 40–70 |
| `BUY_RATIO` | 买入价 / 近两月最高价 | 0.65 | 0.60–0.70 |
| `SINGLE_BUY_MIN` | 单批买入下限 | 1w | — |
| `SINGLE_BUY_MAX` | 单批买入上限（硬约束） | 10w | ≤10w |
| `SWING_TAKE_PROFIT` | 涨卖跌买止盈 | 0.40 | 0.30–0.50 |
| `MOMENTUM_TAKE_PROFIT` | 追涨止盈 | 0.75 | 0.50–1.00 |
| `SECTOR_WHITELIST` | 板块白名单 | 光通信,算力,存储,机器人 | 可增减 |

## 9. 数据模型（增量）

在现有 `Bar` 基础上扩展：

- `Position`：标的、持仓量、平均成本、建仓策略类型（A/B）、建仓时间
- `Signal`：标的、信号类型（买入/清仓）、触发策略、触发价格、参考价（`P_high_2m`）、计算时间
- `TradeAdvice`：标的、方向、金额/仓位、依据规则、风控结论、是否需人工审批
- `TradeLog`（持久状态）：每笔决策的完整快照，供回测与 Harness 演化归因

---

## 10. 里程碑

| 阶段 | 目标 |
|---|---|
| **M1** | 指令层 + 参数表 + 数据模型；白名单选股与最高价/ROI 技能 |
| **M2** | 策略 A/B 信号计算 + 风控门禁 + 交易建议输出 |
| **M3** | 持久状态账本 + 回测框架（验证策略历史表现） |
| **M4** | 子 Agent 拆分（选股/信号/风控）+ 人类审批交互 |
