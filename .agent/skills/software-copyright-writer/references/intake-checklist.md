# 材料收集清单

## 必填输入

- `代码仓库（GitHub / GitLab / Gitee / 本地仓库均可）`
- `官网页面`
- `软著数量`

## 选填输入

- `指定模块`
- `已有截图`
- `Logo/商标注册证`
- `域名备案页`
- `著作权人全称`
- `希望粗拆还是细拆`

## 硬性规则

### 1. Logo

- 如果正文、截图或页面里出现 `Logo`，必须要求用户提供商标注册证。
- 没有商标注册证时，标记为高风险。

### 2. 版权所有

- 文档文字或截图中出现的 `版权所有`，必须与著作权人全称完全一致。
- 不允许缩写、简称、旧公司名或模糊写法。

### 3. 界面截图

- 必须是软件实际运行界面截图。
- 不能使用设计稿、原型图、效果图。
- 图片不要打马赛克，确保 A4 黑白打印时仍可辨认。
- 每篇软著默认准备 `8-12` 张界面截图，优先 `10` 张；截图应覆盖不同功能点，不要重复截同一类页面凑数。
- 截图规划要先按当前软著主题细拆功能，至少覆盖入口、列表/搜索、创建/导入、配置/表单、详情/结果、状态反馈等不同视角中的大部分。
- 优先用 `scripts/fetch_web_evidence.py` 调用浏览器访问产品界面自动截图。
- 需要登录的页面优先使用 `--auth-state` 复用登录态；没有登录态时可使用 `--headed --delay-ms` 人工登录后保存。
- 对需要点击菜单进入的页面，使用 `--input pages.json` 配置 `click_texts`，让脚本按页面文字进入目标界面后截图。
- 对需要长期复用登录态的产品，优先使用 `--profile-dir` 建立持久化 Playwright 浏览器目录。
- 设置 `--ready-text` 作为登录后页面标识；如果未登录，`--headed` 模式会提示用户在弹出的浏览器完成登录后继续截图。
- 默认不要截登录页当产品运行图；如果用户明确强制不登录继续，才使用 `--allow-unauthenticated`，并在材料风险中说明截图可能不完整。
- 截图生成后，导出 `.docx` 时传入 `--evidence-manifest evidence/manifest.json`，让截图自动替换正文中的 `[此处添加对应图片]`。
- 浏览器无法访问或截图失败时，正文中预留 `[此处添加对应图片]`，不要假造界面图。

### 4. 域名

- 页面或截图中出现域名时，必须要求域名备案查询结果页。
- 没有备案页时，标记为高风险。

### 5. 源码

- 提交源码时，优先准备前 `1500` 行和后 `1500` 行，共计不少于 `3000` 行。
- 如果源码总量不足 `3000` 行，则提供全部源码。
- 不要删除注释。
- 不要保留第三方版权信息。
- 源码文件里直接放原始代码，不写摘要、模块介绍、风险说明。
- 源码文档按“文件名 -> 原始代码 -> 文件名 -> 原始代码”的顺序排版。
- 一行源码对应文档里一行内容，不能把多行代码压成一个段落。

### 6. 文字材料

- 目标长度约 `30` 页。
- 目标排版按每页 `30` 行组织内容。
- 目标总字数必须达到 `3w-4w`，通常约 `3.5w`。
- 如果当前正文明显不够这个体量，不能直接交付，必须继续扩写。
- 正文允许插入少量局部代码片段，但只能选当前主题对应的关键代码，不要用大段源码冲字数。
- 正文必须使用正式产品/软件介绍语言，介绍功能、界面、流程、数据处理和运行效果。
- `GitHub`、代码仓库、README、提交记录、分支、clone 等内部分析来源不得出现在最终软著正文中。
- 不要把正文写成“仓库分析报告”或“源码目录说明”，源码路径和文件名主要用于源码原文文件，不作为正文主叙述。
- 标题编号必须严格递进：`## 3.` 下是 `### 3.1`，`### 3.1` 下继续拆标题时必须是 `#### 3.1.1`、`#### 3.1.2`。
- 同级标题必须连续编号，不能跳号；父级标题不存在时不能直接写子级标题。
- Markdown 标题必须顶格写，行首不能缩进；缩进标题容易在 Word 中变成普通正文。

### 7. 名称

- 软著名称必须以 `系统`、`平台`、`软件` 或 `app软件` 结尾。

## 缺口检查模板

按下面格式输出缺口：

```text
当前主题：
缺失材料：
- ...

高风险项：
- ...

继续写作前建议补齐：
- ...
```

## 写作前确认清单

- [ ] 主题名称已经确认
- [ ] 著作权人全称已经确认
- [ ] 主题截图来源已经确认
- [ ] 已规划 `8-12` 张截图，优先 `10` 张
- [ ] 截图清单与当前软著功能细分一一对应
- [ ] 源码范围已经确认
- [ ] 正文字数已经达到 `3w-4w`
- [ ] 正文语言是产品说明口径，不是 GitHub/仓库分析口径
- [ ] 正文未出现 GitHub、仓库、README、提交记录等内部来源表述
- [ ] 标题编号严格递进，符合 `## 3.` -> `### 3.1` -> `#### 3.1.1` 的层级关系
- [ ] Markdown 标题均顶格书写，没有缩进到正文、列表或表格中
- [ ] 正文中的局部代码片段与主题直接相关
- [ ] 源码文件将直接输出原始代码
- [ ] 源码文档将按文件名分段逐行排版
- [ ] 批量输出数量已经确认
- [ ] 是否出现 Logo 已确认
- [ ] 是否出现域名已确认
- [ ] 商标注册证和备案页是否齐全已确认

## 输出动作

材料齐全时：

- 直接进入一次性正文编写
- 进入源码原文整理
- 必要时抓取网页证据

材料不齐全时：

- 先输出缺口清单
- 暂停成稿
