## 扩写要求

- `3. 功能模块设计`、`4. 核心业务流程`、`5. 数据与处理机制`、`6. 关键技术实现` 必须写得最详细，这四章是撑正文厚度的主干。
- 每个模块章节至少写清“目标、功能、输入输出、处理方式、页面表现、模块关系”。
- 平台型主题默认按 `8-12` 张截图规划，优先 `10` 张；模块拆分要细到能覆盖入口、列表、详情、创建、配置、导入、测试、结果、权限或设置、状态反馈等页面。
- 标题编号必须严格递进；例如 `### 3.1 模块一` 下继续拆标题时，必须写成 `#### 3.1.1`、`#### 3.1.2`，不能写成无编号标题或跳号标题。
- 所有 Markdown 标题必须顶格写，标题行首不能有空格、Tab 或全角空格，避免导出 Word 时被当成普通正文。
- 每个流程章节必须写成完整业务过程，不要只写三句空话。
- 平台型主题一定要把模块协同关系写出来，不然看着就像几个功能散装拼一起。
- 整体正文字数必须控制在 `3w-4w`，通常约 `3.5w`，低于 `3w` 继续扩写，高于 `4w` 就收束冗余内容。

## 写作要求

- 平台型主题要写清模块协同关系。
- 不要堆空话，尽量落到真实流程和真实页面。
- 如果多个主题来自同一仓库，必须确保这里的流程和模块描述不与其他主题大面积重叠。
- 正文必须使用正式软著产品说明口吻，围绕“本软件/本系统/该平台”的功能、流程、界面、数据处理和运行效果展开。
- 正文中不得出现 `GitHub`、`仓库`、`repo`、`README`、`开源地址`、`提交记录`、`分支`、`clone` 等代码托管或代码分析口径。
- 不要写“通过分析仓库发现”“本仓库包含”“GitHub 页面显示”等来源性表达；仓库只用于内部理解和准备源码原文。
- 必须扩写到足够支撑“每页 `30` 行、约 `30` 页、总字数 `3w-4w`”的申报体量，不能写成缩水版说明。
- 可以加入少量局部代码片段，但必须服务于当前主题说明，不能拿大段代码凑字数。
- 信息齐全后直接一次性输出完整正文，不要先给目录草稿、提纲或分段试写。
- 截图优先调用浏览器访问产品界面自动生成；每篇默认准备 `8-12` 张、优先 `10` 张，暂时无法补齐时，直接保留占位符 `[此处添加对应图片]`。
- 正文字体要求：中文 `宋体`，英文和数字 `Times New Roman`，`12pt`，`1.5 倍` 行距。
- 标题字体要求：
  - 一级标题：`黑体`、`16pt`、加粗
  - 二级标题：`黑体`、`14pt`、加粗
  - 三级标题：`黑体`、`12pt`、加粗
