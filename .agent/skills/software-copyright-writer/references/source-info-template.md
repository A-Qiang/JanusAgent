# 源码原文文件规则

虽然文件名保留为 `source-info-template.md`，但这里的实际用途已经改成“源码原文文件规则”，不是说明文模板。

## 核心要求

- 每个软著必须生成一份独立的源码原文文件。
- 文件里直接放原始代码，不写概述、不写模块说明、不写风险分析。
- 保留原始注释。
- 不保留第三方版权信息。
- 总行数优先满足前 `1500` 行和后 `1500` 行，共计不少于 `3000` 行。
- 如果对应代码总量不足 `3000` 行，则直接提供全部源码。
- 按“文件名 -> 原始代码 -> 文件名 -> 原始代码”的顺序组织文档。
- 导出 Word 时，一行源码必须对应一行文档内容，不能把多行代码压成一个段落。

## 推荐输出格式

优先输出纯文本源码文件，例如：

- `xxx-源码原文.txt`
- `xxx-源码原文.md`

如果后续需要正式文档，再导出：

- `xxx-源码原文.docx`

## 内容规则

- 可以按已经确认的源码范围收集多个文件内容。
- 默认只收集真正的源码文件，不收 README、说明文档、构建产物和第三方目录里的垃圾内容。
- 默认按路径顺序稳定拼接原始代码。
- 文件名单独占一行，再接对应原始代码。
- 不需要在代码之间插入解释段落。
- 不需要写“源码范围说明”“模块与功能映射”“风险备注”。

## 最小模板

源码文件允许只有代码内容本身。
如果确实需要最小化标识，只保留文件路径分隔，不要增加解释性文字。

```text
===== 文件名: src/module/example.py =====
[原始代码逐行内容]
```

## 生成方式

优先使用：

```bash
python3 scripts/assemble_source_code.py \
  --path repo/src/module-a \
  --path repo/src/module-b \
  --output out/source-code.txt
```

## 禁止事项

- 禁止把源码文件写成摘要文档
- 禁止只写目录说明不贴代码
- 禁止删除注释后再输出
- 禁止混入“该模块主要负责……”这类解释文字
- 禁止把 `README.md`、文档、构建产物、锁文件、第三方依赖目录一股脑拼进去
