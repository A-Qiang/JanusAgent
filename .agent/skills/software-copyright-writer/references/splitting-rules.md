# 主题拆分规则

## 目标

把一个仓库拆成“可以独立申报”的软著主题，而不是把一个大仓库生拉硬拽成几篇名称不同、内容高度重复的废稿。

## 拆分前判断

先判断仓库是否具备这些条件：

- 有多个相对独立的功能域
- 每个候选主题都能找到对应代码范围
- 每个候选主题都能找到对应界面或页面证据
- 每个候选主题都能写出明显不同的业务目标和流程

如果以上条件不成立，不要强行拆分。

## 粗拆

适合场景：

- 用户只要少量软著
- 仓库更像平台型产品
- 子模块之间耦合较强，但还能区分主能力域

常见粗拆方式：

- 按大功能域拆
- 按角色工作台拆
- 按平台子系统拆

粗拆输出必须说明：

- 每个主题覆盖哪些主能力
- 哪些子功能被纳入该主题
- 为什么不再继续细拆

## 细拆

适合场景：

- 用户要求的数量更多
- 指定模块本身体量较大
- 模块内部包含多个独立流程或子系统

常见细拆方式：

- 按业务流程拆
- 按模块职责拆
- 按场景闭环拆

细拆输出必须说明：

- 每个主题的最小边界
- 与其他主题的差异点
- 是否存在内容重叠风险

## 主题评估维度

给每个候选主题逐项判断：

| 维度 | 判断问题 |
|---|---|
| 业务目标 | 这个主题是否有单独目标？ |
| 功能闭环 | 是否能独立讲清输入、处理、输出？ |
| 页面证据 | 是否能找到实际运行界面或官网证据？ |
| 源码边界 | 是否能落到清晰目录、模块或关键文件？ |
| 文案差异 | 写出来后是否能与其他主题明显区分？ |

如果同一个候选主题在大多数维度都答不上来，直接淘汰。

## 命名规则

- 名称必须体现主题核心能力
- 避免“综合管理系统”这种啥都能套的空名
- 结尾必须是：
  - `系统`
  - `平台`
  - `软件`
  - `app软件`

## 不合理拆分的典型信号

- 只是把一个页面的不同 Tab 强行拆成多个软著
- 只是把同一流程的前后步骤拆成多个名字
- 只是换了同义词，正文内容还得大量复用
- 没有独立截图和独立源码范围

## 用户确认输出模板

```text
推荐拆分方式：粗拆 / 细拆

候选主题 1：
- 名称：
- 边界：
- 主要功能点：
- 页面证据：
- 源码范围：
- 与其他主题差异：

候选主题 2：
- 名称：
- 边界：
- 主要功能点：
- 页面证据：
- 源码范围：
- 与其他主题差异：
```

## 数量不合理时的回复方式

```text
当前仓库可独立申报的主题数量不足以稳定支撑你要求的数量。

原因：
- ...

建议：
- 降低软著数量
- 指定一个更大的模块再细拆
- 补充更多实际运行界面和代码边界信息
```

## 批量生成规则

如果用户直接要求：

- `给我产 5 份软著`
- `直接生成 5 份`
- `按这个仓库直接跑五份文档`

且仓库体量、材料和主题边界已经足够支撑对应数量，则不要一份一份停下来确认，直接连续生成全部文档。

只有在下面这些情况才允许停下来：

- 仓库无法支撑目标数量
- 模块边界不清楚
- 材料明显缺失
- 多个主题会高度重叠
