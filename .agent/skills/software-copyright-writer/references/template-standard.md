# 标准版软著正文模板

适用于功能边界清晰、单主题体量适中的软著。

这个模板不能只当章节目录来用，必须把每一章写实、写满、写到能支撑申报页数。少量 bullet 只能辅助说明，主内容要用完整段落展开。

```markdown
# {软件名称}

## 1. 软件概述

### 1.1 建设背景
- 说明行业背景、业务场景、现有问题和建设必要性。
- 至少展开 2-3 个自然段，不要只写一句话。

### 1.2 开发目标
- 说明软件建设目标、预期效果、解决痛点和使用价值。
- 需要写清楚为什么要建设这个软件。

### 1.3 适用范围
- 说明适用对象、应用组织、部署范围和主要使用角色。

## 2. 总体设计

### 2.1 系统定位
- 说明该软件在整体业务中的定位和主要职责。

### 2.2 总体架构
- 说明前端、后端、数据存储、接口交互、部署方式。
- 不能只报技术栈，要写清组件协同关系。

### 2.3 数据流与处理思路
- 说明数据从输入到处理再到输出的流转方式。

## 3. 功能模块设计

### 3.1 模块一
```

## 扩写要求

- 每个二级标题下至少写 2-4 个自然段，不要整页都是短 bullet。
- `3. 功能模块设计` 和 `4. 关键业务流程` 必须写得最充实，这是正文撑页数的主力，不是拿来打卡的。
- 功能模块要细拆到可以支撑约 `10` 张截图，优先按入口、列表、详情、新建、编辑、上传、配置、结果、设置、状态反馈等角度组织。
- 标题编号必须严格递进；例如 `### 3.1 模块一` 下继续拆标题时，必须写成 `#### 3.1.1`、`#### 3.1.2`，不能写成无编号标题或跳号标题。
- 所有 Markdown 标题必须顶格写，标题行首不能有空格、Tab 或全角空格，避免导出 Word 时被当成普通正文。
- `5. 关键技术实现` 不能只写技术名词，必须说明实现思路、作用和处理逻辑。
- `7. 界面与交互说明` 要围绕真实页面写说明，每篇默认准备 `8-12` 张截图、优先 `10` 张；截图缺失时再留 `[此处添加对应图片]`。
- 整体正文字数必须控制在 `3w-4w`，通常约 `3.5w`，低于 `3w` 继续扩写，高于 `4w` 就收束冗余内容。

## 写作要求

- 用陈述句，不要写成营销软文。
- 用正式软著产品说明口吻介绍“本软件/本系统/该平台”的功能、流程、界面和运行效果。
- 模块说明尽量贴近实际代码和页面。
- 正文要能和截图、源码原文相互印证。
- 正文中不得出现 `GitHub`、`仓库`、`repo`、`README`、`开源地址`、`提交记录`、`分支`、`clone` 等代码托管或代码分析口径。
- 不要写“通过分析仓库发现”“本仓库包含”“GitHub 页面显示”等来源性表达；仓库只用于内部理解和准备源码原文。
- 必须扩写到足够支撑“每页 `30` 行、约 `30` 页、总字数 `3w-4w`”的申报体量，不能只写几页概述。
- 可以加入少量局部代码片段，但必须紧贴主题功能和技术实现，不能拿大段代码灌水。
- 信息齐全后直接一次性输出完整正文，不要先给目录草稿、提纲或分段试写。
- 截图暂时无法补齐时，直接保留占位符 `[此处添加对应图片]`。
- 正文字体要求：中文 `宋体`，英文和数字 `Times New Roman`，`12pt`，`1.5 倍` 行距。
- 标题字体要求：
  - 一级标题：`黑体`、`16pt`、加粗
  - 二级标题：`黑体`、`14pt`、加粗
  - 三级标题：`黑体`、`12pt`、加粗
