# 周报撰写 Skill（STAR 法则）

## 使用场景

当用户要求撰写周报、工作汇报、阶段性总结时触发。核心目标是按 STAR 法则把"做了什么"和"产生了什么价值"讲清楚，输出结构化 Markdown 周报。

## 输出模板（最终交付物）

生成的周报文件默认命名为 `周报-YYYYMMDD.md`，保存到用户指定目录（未指定则询问）。结构如下：

```markdown
# 周报（YYYY-MM-DD ~ YYYY-MM-DD）

## 项目投入
- 智能销售 90% + 维修助手 10%

## Part 1 周总结

### 1. [需求标题：用产品语言描述价值]

【本周进展】
[Situation + Task + Action：为什么要做、推进了哪些任务、具体动作]

【结果总结】
[Result：对项目 / 客户 / 团队产生的价值]

### 2. [下一个需求标题]

【本周进展】
...

【结果总结】
...

## Part 2 价值焦点思考

[开放性内容：本周工作对客户价值、部门/公司战略、本职成长的贡献。建议 2-3 个小主题分点展开，每个主题一句结论 + 一段展开]

## Part 3 下周计划

### 1. [最有价值的一项工作]

- **目标**：为什么做，追求什么结果
- **做法**：如何做，关键子任务列表
- **资源**：需要什么支持或输入

### 2. [第二项工作，可选]
...
```

## 执行流程

严格按以下三阶段推进，不要跳步。每阶段结束前显式向用户确认。

### 阶段 1：收集工作内容

按以下顺序并行收集素材，形成"原始工作项清单"：

1. **Git 提交记录**：默认取最近 7 天
   ```bash
   git log --since="7 days ago" --author="$(git config user.name)" --pretty=format:"%h | %ad | %s" --date=short
   ```
   若用户指定其他时间范围（如"上周"、"最近两周"），调整 `--since`。多仓库场景需在每个仓库分别执行。

2. **Workspace 相关文档**：扫描本周新增/修改的文档
   - `docs/`、`.agent/`、`plans/`、`rfcs/`、`survey/` 等目录下的 Markdown 文件
   - 使用 `git log --since="7 days ago" --name-only --pretty=format:` 提取本周变更过的文件路径
   - 重点读取 RFC、调研、方案类文档，提炼"推进了什么"

3. **访谈用户消除 Gap**：git 与文档无法覆盖的部分（如线下会议、跨团队沟通、未提交的探索）通过提问补齐。使用 AskUserQuestion（若可用）或对话式提问，一次最多 3-4 个问题：
   - 本周是否有未落地为 commit / 文档的重要工作？（如会议决策、方案讨论、口头对齐）
   - 每项工作对应的**项目归属**是什么？（智能销售 / 维修助手 / 其他）
   - 每项工作的**结果与价值**是否已明确？如未量化，是否有近似指标？
   - 下周最优先要推进的一项工作是什么？

**输出物**：一份"原始工作项清单"，字段包括 `工作项 | 项目归属 | 相关 commit/文档 | 完成度 | 结果 | 待澄清项`。展示给用户确认后再进入阶段 2。

### 阶段 2：内容整理与归纳

将原始工作项清单按"需求维度"聚类归纳：

1. **合并同一需求下的多个 commit / 文档**：一个需求对应一个周总结条目，不要按 commit 罗列。
2. **提炼产品语言标题**：让产品/上级一眼看出"这个需求是什么、对项目意味着什么"。避免技术黑话式标题。
   - 好：`故障诊断标准化标注体系建立与基准测试集构建`
   - 差：`修复 label 字段 bug`、`refactor eval pipeline`
3. **按 STAR 拆分为"本周进展 + 结果总结"两段**：
   - 本周进展 = Situation（为什么做 / 背景痛点） + Task + Action（推进的任务与具体动作）
   - 结果总结 = Result（对项目的价值：解决了什么问题、带来什么能力、量化指标）
4. **优先级排序**：把产出价值最高的需求排在前面，一份周报建议 3-6 项。

**输出物**：整理后的"周总结草稿"（Part 1 内容），展示给用户确认。

### 阶段 3：生成周报文档

1. 按上文模板填充四部分：项目投入、Part 1 周总结、Part 2 价值焦点思考、Part 3 下周计划。
2. **价值焦点思考** 从阶段 2 的结果中萃取 2-3 个战略层面的主题（如"基建价值"、"方法论沉淀"、"跨团队协作"），每个主题一句结论 + 一段展开论证。避免复述 Part 1。
3. **下周计划** 从访谈获得的优先级出发，第一项必须写清"目标 / 做法 / 资源"三要素并列出子任务。
4. 用 Write 工具生成周报 Markdown 文件，路径默认 `周报-<本周周五日期>.md`（未指定则询问用户）。
5. 生成后简要摘要展示给用户，询问是否需要调整。

## STAR 撰写范例（对照参考）

```markdown
### 1. 故障诊断标准化标注体系建立与基准测试集构建

【本周进展】
为了建立科学的准确率评价体系，本周输出了《故障诊断标注规范》，明确了知识抽取的标注字段与标准。同步完成了 30 个典型测试 Case 的精细化筛选，并完成了对原始故障案例文档的结构化解析与关键字段抽取。

【结果总结】
确立了衡量 80% 准确率目标的"标准（Ground Truth）"。该标注集的建立解决了此前评估维度不统一的问题，使算法迭代具备了量化的对标基准。
```

**关键要点**：
- 标题包含"做了什么 + 项目意义"两个信息（体系建立 + 基准测试集）。
- 【本周进展】以"为了……本周输出了……同步完成了……"结构展开，因果与动作齐全。
- 【结果总结】给出可对标的价值语言："确立了 Ground Truth"、"解决了 XX 问题"、"使 XX 具备 XX 能力"。

## 价值焦点思考撰写范例

```markdown
本周在 Agent 离线工程化和多 Agent 评估上的突破，对团队长远发展具有战略性的"基建价值"：

**解耦带来的敏捷性**：此前算法迭代往往受限于业务系统的发版节奏和环境限制。本周实现的代码解耦，意味着我们拥有了独立的"练兵场"，算法团队可以在离线环境中高频、低成本地进行工具链调整和 Prompt 优化，验证通过后再同步至生产。

**评估 agent 的搭建与有效性验证**：大模型评估最大的痛点在于其不确定性。本周构建的多 Agent 投票评估机制，本质上是利用 test-scaling 的思想，用算力换性能，用多个 Agent 的共识抵消个体的幻觉和噪声。
```

**关键要点**：先给整体结论（战略性的"基建价值"），再用**加粗小主题** + 一段论证展开，每个主题独立成立。

## 下周计划撰写范例

```markdown
### 1. 开展长尾故障教材补充（P0 级补盲）

- **目标**：针对进水、落水等 A 类长尾 Badcase，补充专项专家经验教材。消除已知知识盲区，追求 Pass@1 的进一步提升。
- **做法**：
  - 从 Badcase 池筛选 A 类 case 20 例
  - 联合领域专家共创教材草稿
  - 灌入知识库并回归测试集验证
- **资源**：需要 2 位维修领域专家配合 2 小时访谈
```

## 关键约束

- **一个需求一个条目**：不要按 commit 罗列，避免琐碎化。
- **结果必谈价值**：禁止"完成了 XX"式的流水账结果，必须回答"这对项目意味着什么"。
- **产品语言写标题**：标题让非技术读者也能理解需求价值。
- **项目归属显式化**：每项工作明确归属"智能销售"或"维修助手"，与 Part 0 项目投入呼应。
- **访谈优先补 Gap**：git 和文档能覆盖的不用问，只问机器无法推断的部分（会议、决策、优先级、未量化的价值）。
- **不虚构**：所有工作项必须有 commit / 文档 / 用户口述三者之一作为来源，否则不写入周报。

## 反模式（避免）

- ❌ 直接把 git log 粘贴成周总结
- ❌ 标题使用 `refactor xxx`、`fix bug` 这类技术描述
- ❌ 结果段只写"完成了标注工作"，不谈价值
- ❌ 价值焦点思考复述 Part 1 内容
- ❌ 下周计划只有目标没有做法与资源
- ❌ 未与用户确认清单就直接生成最终文档
